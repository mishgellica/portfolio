the challenge

What problem did we solve?

Traditional HR systems are full of security holes — privilege escalation, unauthorized employee record access, data tampering, and leaks of sensitive personal and financial data. Most of these aren't theoretical risks either; they happen when systems are built without thinking about access control from the ground up.

what we built

How did we solve it?

We developed a multi-tier HR web architecture protected by rigid Role-Based Access Control levels — Admin, HR Manager, and Employee. We conducted a full Security Threat Analysis to identify system vulnerabilities, then implemented audit logging, input sanitization, and proactive security patches to actively mitigate data disclosure and unauthorized manipulation risks.

what i learned

What did this teach me?

Security used to feel like an afterthought to me — something you bolted on at the end. Building IrisHR forced me to think about it as a structural decision made at every layer. When you're handling employee salaries, performance reviews, and personal identifiers, the cost of getting it wrong isn't just a bug you patch — it's someone's data being exposed. Learning to model threats before writing a single line of code changed how I approach every project now. I also got deep into RBAC implementation and learned how nuanced permission systems really are — the difference between what a user should see and what they can technically access is a gap you have to deliberately close at every endpoint, not just the frontend.
